00 秒杀微服务项目 笔记

2024-05-07T04:46:52.000Z2025-01-15T10:15:21.315Z

秒杀微服务

秒杀微服务的优势：

  1. 高并发系统
  2. 高可靠系统
  3. 适合微服务架构
  4. 容易版本迭代
  5. 核心矛盾聚焦，容易吃下
  6. 场景脍炙人口
  7. 涉及组件多，技术栈全面

秒杀服务流程

  • 秒杀前
    1. 首先你是一名秒杀客户，你进入了商品详情页，此时其实就拉取到了商品的信息
    2. 当你发现时间没到显示倒计时，那就等到时间差不多再来
    3. 倒计时结束，这时候其实可以猜测每次进入当前的秒杀页面，后端都会按例检查下是否已经秒杀结束，因为秒杀如果已经都付完款结束，就没必要再进行了
  • 秒杀开始到秒杀获胜
    1. 检查是否名额已经被秒完，是的话暂时不能点购买，这里注意，名额秒完不代表秒杀结束，因为这时候不是完全没机会，说不定还有人不付款，或者因为系统原因有少卖的名额退还
    2. 如果还有名额，你就激动地点击购买，这时候就是有希望了，这个羊毛薅定了
    3. 进行库存抢夺，这一步有很多设计点，这里初步认为就是直接抢一个资格，具体实现下可能会更复杂，比如会分为预扣库存和扣减库存，这是秒杀的核心
    4. 抢到了就可以创建订单，离成功不远了
  • 秒杀获胜后还得付款
    1. 在时间内支付，如果超过时间没有支付，那么这笔交易就取消，并且库存会收回
    2. 购买成功，准备收货

技术层面来说：

  1. 倒计时，第一反应看到的是 瞬时流量，一个秒杀活动的流量可以说 99% 以上会在活动开始前这一瞬间打入，这意味着我们面临的是一个 高并发场景，而且是一个瞬时高峰场景，这一点上我们其实很容易想到 消息队列削峰
  2. 还是倒计时，除了瞬时流量之外，我们还应想象出成千上万用户在屏幕前等待倒计时，那一刻的激动心情，如果服务崩了，这不得炸了？所以我们应该看到 稳定性，秒杀服务对稳定性的要求很高，其他服务可能偶尔挂一下，还能接受，但秒杀这种就是冲某个时间点的一波流，挂几分钟多伤啊
  3. 购买，涉及到购买，那其实就是交易场景，哪怕你秒的是一些数字化的东西，比如优惠券，它也是有价值的，我们都可以看作是商品，对这种商品交易场景，我们应该看到 可靠性，一旦发生交易数据的不一致问题，比如多卖，比如少卖，都会带来恶劣的影响，这方面的处理措施也是要考虑的

架构设计流程

高并发

高并发的优化思路

  1. 缓存优化
    • 通过将数据临时存储在高速存储中，以 提高查询性能，注意是查询性能，所以缓存是优化读操作的。不同层面都可以有缓存，比如数据库缓存，比如返回包缓存，当然也可以按缓存所在位置来区分，远端缓存和本地缓存
  2. 群策群力
    • 一个节点扛不住请求，那么就多个，这种思路的例子有很多，比如应用层服务水平扩容，比如数据库分库分表，以及 Redis 集群化方案这些都算
  3. 流量限制
    • 通过各种手段减少流量，比如风控提高门槛，比如设置值超过就拒绝，这些流量限制的方式其实本质是降低实际请求量。
  4. 池化技术
    • 池化技术说白了是资源复用技术，即将用了的资源，先不释放，再次利用这份资源服务新的操作或请求，池化技术 包括 数据库连接池、HTTP 连接池、内存池、线程池 等等
  5. 读写分离技术
    • 顾名思义，将读操作和写操作分离开来，以提升性能，比如如果读多写少场景，对数据库的写操作势必也会被读操作所影响，要提升写操作的性能，就可以做读写分离
  6. 索引优化技术
    • 不管是什么组件，走索引大多是比不走索引快得多的，热点查询都要尽可能地去添加索引并且尽可能选择更合适的索引优化以提高性能
  7. 缓冲技术
    • 使用更高速的组件，预先承载流量，再慢慢传递到低速组件，比如我们的秒杀场景，就可以通过更高速的 Redis 做名额预扣，再传递到底层存储，当然使用消息队列削峰其实也是缓冲技术
  8. 预加载技术
    • 主要针对一些活动，就拿我们秒杀活动举例，如果等活动开始再将库存资源加载，是不是就可能来不及，也可能因为加载操作又拖慢了系统的性能，这时候我们就可以将一些资源预先加载，这里包括加载 MySQL 数据库，也包括加载 Redis缓冲
  9. 聚合写技术
    • 如果是大规模写入场景，这时候其实可以考虑聚合写入，也就是将多次操作合并为一次操作，通常的思路有 2 个，第一个是仅仅单纯多条语句拼装在一起，发向存储，这种方式只是减少了网络 I/O 成本，另一个思路是通过计算聚合，比如用某个 Key 计数，10条请求每个请求都是 num+1，那么合并在一起，不就是 num+10 了么

高并发的应对思路

  • 流量限制
    1. 风控，接入风控系统，增加购买的门槛
    2. 预约，提前预约，按经验，预约流量起码小 1 个量级
    3. 验证码答题，拉平请求，原来通过点击，请求在 1s 内全来了，通过验证码，大家操作的时间就被拉到 1-10s 之间，请求相当于就分摊了
    4. 设置阈值，设定流量的最大阈值，超过阈值的请求直接拒绝这四种方式都属于限制流量的范畴，前三种其实更多是属于产品业务上的优化，设置阈值则是比较通用的技术做法，总会有一个流量上限，不可能无限扩充资源
  • 异步化削峰
    • 秒杀还有个思路是异步化削峰，落地到技术中实际就是缓冲技术，下面看一下流程进行异步化
      1. 第一种方式是，全流程异步化。也就是在对请求做完参数检查、频率限制之后，把后续的一整个流程进行异步化处理。但这样会产生一个大家可能都经历过的用户体验问题：秒杀后菊花转了很久，最后弹了个框提示：奖品已抢光
        • 很多产品不太可接受这样的用户体验，所以如果有其他的办法，尽量选择别的，
      2. 第二种方式，库存扣减异步化，引入通过 Redis 来预扣库存，这里为了避免混淆，我们把 Redis 中的信息定为名额。抢到名额的，大在绝大部分情况下都能发券成功时，再让用户等待一会是可以接受的
        • 第二种方式也是更普遍的方式，我们后面的代码也是按第二种方式来实现。这里再展开一下业务逻辑：
          1. 校验请求，即检查请求的参数对不对，检查用户是否有资格秒杀对应商品
          2. 确认并扣减名额
          3. 记录扣减名额信息
          4. 将扣减名额成功的信息直接发给库存服务或者先丢入异步队列
  • 聚合写
    • 聚合写也是秒杀系统中可以应用的一个技术，主要体现在最终对 MySQL 库存操作的时候，比如：对某个商品，扣减1000次库存，就是1000次请求，其实是合并成一次请求，这一次请求里面的库存数量直接减1000即可
    • 实际上，如果已经引入了一些削峰手段，除非你的业务规模真的特别庞大，不然这里的聚合写仅属于锦上添花的做法，很多团队没遇到瓶颈之前也不会去做，避免增加系统复杂度

高精准

高精准问题：就是要数据的准确度很高，秒杀是商品交易行为，这种业务对精准度的要求很高，也就是尽可能最终其实是跟钱挂钩，说多少就是多少，比如一开始说秒杀 1000 个，那么最终秒杀成功 1001 个，或者只能秒杀成功 999 个，这都是应该避免的。除了库存之外，对用户的限购也应该精准

一定不要多买

如何限制多卖呢？

  • 核心来讲需要严格控制库存，通常需要一致性强的组件，比如：MySQL，通过其持久性保证数据不丢失，同时在扣减时做好可重入设计，可以通过 where 条件做好乐观锁保证

尽量不要少卖

少卖主要由于以下两个原因：

  1. 库存扣减了，但订单生成失败
  2. 订单生成了，用户没付款，而库存回退逻辑又失败了

名额加载回Redis：

  • 这里需要判断 MySQL 库存还存在，并且 Redis 名额已经没有了，此时可能就是少卖了，但是还有一种可能性是请求还没到达 MySQL，为了避免还有请求在处理中，一般可以增加一个逻辑：上述状态维持超过 x 分钟（比如：5分钟），分钟级的时间足够消息处理了

超时库存退回MySQL：

  • 定时任务扫，如果订单超时未支付，那么就把额度加载回 MySQL
  • 注意这种退回额度加载，一定是要避免多次回退库存到 MySQL 的，还是可以通过 乐观锁 来保证

  • 限购一般在 Redis 就可以做，但是 Redis 本身确实没 MySQL 那么可靠，所以最终用户额度还是要放到MySQL，所以这里需要 MySQL 的额度配置表，当然除了维度全局的额度，还可以支持用户级别的额度，实现更精细的控制

高触达

高触达：也就是能触达到真实的客户，只有这样才能产生传播效应，毕竟做活动的目的就是让利于真实的用户，通过活动热度打开市场

限购

  • 限制单个用户能买的数量，比如：很多秒杀活动就是一人能秒1件，如此一来，黄牛就得搞更多的账号来操作，这样可以增加黄牛作恶的成本，这种方式直接且有效，一般都会被采用
  • 在技术实现上，限购可以分为 全局限购，和 特定用户限购，全局限购是所有用户共用的限购策略，局部限购则是特别的用户定制化的限购策略，这些策略可以通过 MySQL 配置。

限IP

  • 限 IP 流，部分黄牛是用机器人来刷的，这时候可以针对 IP 做限制，比如：如果一个IP地址，每秒的请求量大于100，人类的点击速度不可能这么快，那么这绝对是有问题的，这时候就可以对相关 IP 进行限制。当然，也许你会说 IP 是可以伪造的，这没有问题，我们进行 IP 限制主要还是增加作恶成本，网络攻防没有绝对，都是你来我忘，这种方式可以将一部分技术或者工具比较落后的黄牛拦在门外

验证码

  • 体验感比较差

风控

  • 通过风控系统，在秒杀开始前及时发现异常账号，提高秒杀的门槛，这种一般是接入现成的风控系统，不会自己单独做一套，要么是公司里有专门的团队提供支持，要么就是购买第三方服务

实名制绑定

  • 实名应该是提前去做的

架构部分

层次结构

  • 秒杀结构可以分为如下几层：
    1. 接入层，接入层包含了负载均衡和网关服务
    2. 服务层，即秒杀服务及关联的其他业务业务，比如：商品服务、订单服务，当然还有用户服务这类非常独立的业务领域服务
    3. 存储层，MySQL、Redis 这类存储，除此之外其实还有涉及到信息流转的消息队列

上下游服务

秒杀服务的上下游关系：

  1. 当发起一个秒杀请求，这个请求首先到达 负载均衡，负载均衡 会转发到 合适的网关服务，网关服务做了鉴权之后，再传递到我们的业务层，这里就是秒杀服务
  2. 秒杀服务收到请求之后，首先会检验参数，参数有问题就直接拒绝，没问题会先查看商品信息，这里其实应该是调用商品服务，我们为了简单就是直接查询商品表
  3. 商品信息没问题的话，就会进行库存扣减，扣减成功这时候就需要跟订单服务进行交互，生成一个订单了，有了订单之后是不是就涉及到支付服务了，当然这里我们为了聚焦矛盾、也为了简化一点，订单逻辑是写在秒杀服务里的，支付服务也省略了

服务交互

对外的接口提供 HTTP 接口，对内的接口提供 RPC 接口，比如：秒杀请求是用户直接发起，那么就是 HTTP 接口，如果内部之间需要交互，比如秒杀服务向订单服务发请求，这时候就是走 RPC 接口

服务注册发现

  • 一个服务怎么知道另一个服务的地址呢，特别是我们的服务都是分布式的，不可能写死多个地址吧。微服务下的解决方案就是服务注册与发现
  • 每个服务进程上报自己的信息给服务发现组件，比如 ETCD、比如 NACOS，进程会将自己的 服务名字 和主机 ip 和 端口 上报到服务发现。
  • 其他服务要访问这个进程之前，就先通过服务名从服务发现拿找到请求的地址，再进行请求，这样只要有个访问的名字，无论你增加多少台机器，都是可以找到的，具体用哪个地址，可以有不同的算法，比如：轮询访问，还有随机访问

存储层

后台服务除了业务处理，最重要的就是存储，数据总要有个安身之处，这里秒杀服务为了可靠性会使用 MySQL 做底层存储，为了性能，会使用 Redis 做名额存储

存储选型

关系型数据库选型

  • 一般常见的有 MySQL、Oracle、SQLServer、TDSQL。关系型数据库普遍具备如下优点：
    1. 可靠性强，数据是持久化到磁盘，没有丢失数据的风险。
    2. 结构规范清晰，每一列数据都有格式和长度规范，一个表中每一行数据的属性都是相同的
    3. 支持事务，很多时候多个操作希望要么成功、要么失败，比如：我们扣减库存和记录库存扣减事件这两个操作，我们不希望只有一个成功，而是希望他们成对出现
  • 关系型数据库两点核心的、没有歧义的缺点：
    1. 关系型数据库性能普遍都不高，这个和按行存储、维护索引、事务都有关系，可以理解为因为太过规范可靠，所以速度自然快不起来
    2. 表结构扩展非常不便，特别上线之后，增减字段都会对线上产生影响

非关系型数据库

  • 这里指以 键值对 形式存储的非关系型数据库，比如：Redis、Memcache
  • 这类 NoSQL 优点其实就是 快，基于 内存 存储，KV 型结构，这自然快得飞起，利用 Redis 自带的 BenchMark 做测试,读性能可以 10W+，写性能也是 7，8W
  • 这类 NoSQL 缺点也很明显，最核心的是可靠性不强，可能会丢失数据，如果数据比较重要那么就不适合用 NoSQL 做底层存储

秒杀要存的数据类型

首先，我们要明确要存储的是什么数据：

  1. 库存数据
  2. 秒杀记录数据，秒杀之后一般会有个记录的
  3. 订单记录，记录订单的信息，在大公司可能会有单独的订单系统，小公司可能就是写在业务中

库存数据、秒杀记录数据，都是需要是比较精准的，也需要是持久化的，同时也是需要事务的，所以需要比较可靠的数据库

数据库 运行平台 扩展性 性能 价格
MySQL Linux, Windows 单机 几千级 正常
Oracle Linux, Windows 单机 几千级 贵
SqlServer Windows 单机 几千级 正常
TDSQL Linux 分布式 分布式可扩展 昂贵
  • 这里很明显，后端服务还是跑在 linux 上比较好，SqlServer 是不会考虑的。至于 Oracle 更多用于工业场景，互联网主流的还是 MySQL
  • TDSQL 可以看作是腾讯的分布式 MySQL，功能和 MySQL 大部分一样，也完全能支持我们的场景，但就是非常贵，一般来说除非有特殊需要，中小团队也不会主动去接入 TDSQL。所以这里我们最终选择MySQL，这个互联网最流行可靠的关系型存储，来做我们的底层数据库

高并发下的辅助存储

  • 用 NoSQL 粗略地做一遍秒杀筛选，大多数时候拿到名额的人最终都能拿到库存，但也有概率有一定偏差，库存最终还是受强一致的 MySQL 保护
  • 对比下 Redis 和 Memcache 的优缺点：
    • Redis 数据结构比较丰富，而 Memcache 支持只最简单的字符串结构
    • Redis 支持持久化，Memcache 不支持持久化，所以 Memcache 如其名一样，真的只适合做缓存
    • value 数据比较大时候，Memcache 性能是优于 Redis 的
    • Redis 比 Memcache 更流行，基本所有后端团队都会应用 Redis

表结构设计

库存表

库表表：记录库存库存信息

表名：t_seckill_stock

字段 类型 说明

幂等性设计原则

幂等性：简单来说，即同样的操作无论执行多少次，结果都一样，也就是说只有第一次造成了数据的改变

  • 拿转账举例，同一笔转账请求，即使因为网络波动或者失败重试重发了多次，但最终实际的金额流转只会有一次

幂等实现方式

  • 依托唯一Key
    • 可以看到通过唯一 Key 限制，是不能重入插入数据的，并且是有鲜明的报错
  • 依托状态
    • 状态更新覆盖，即通过状态来判断，如果状态已经变了，则影响行数为 0
  • 依托额外记录
    • 可以单独做一个表，可以理解为请求记录表，即通过记录请求 ID 来实现不可重复插入

秒杀中幂等考虑

秒杀接口底下所有逻辑，都是要幂等的，可以分成多个层面来看：

  1. 预扣名额逻辑，需要支持幂等，这里需要 Redis 来做，所以 Redis 里面需要有秒杀记录来支撑，当然Redis 依托唯一 Key 的方式和 MySQL 不一样，不是通过不可重复插入，而是直接 Lua 判断即可，因为Redis 是单线程的
  2. MySQL 库存扣减逻辑，在 MySQL 这一层会比较清晰，库存表可以通过库存字段做乐观锁，秒杀记录表可以通过 secNum 字段做唯一 Key 重复判断
  3. 另外就是消费消息队列需要幂等，扩展一下就是要做到只消费一次，这个需要消费者消费成功才 ACK 避免不丢数据，外加前面说到的 MySQL 幂等性扣减库存保证重复执行是幂等的

异常情况容错分析

同一个用户对同一个商品多次秒杀

  • 正常来说，一个用户对一个商品只要有名额，就可以多次秒杀，但这里要注意，不能同时多次秒杀，也就是说如果一个秒杀请求还在流程里，那么此时是不应该让他能再进行另一个秒杀请求的
  • 所以这里的解决方案就是 加这么一个限制，进入秒杀流程时会判断这个用户是有具备秒杀号，如果具备了就返回他秒杀号，否则才会往后面走

名额扣减了，但是记录没写进去

  • 这里说的名额是指 Redis 预扣逻辑里的名额，这种异常情况出现在，Lua 执行一半，突然 Redis 挂掉了，Lua 本身是没原子性的，他只是依赖 Redis 的单线程逻辑来实现一定的原子性，保证一个 Lua 操作不会被其他操作中断，但是挂了就没办法了
  • 这种情况导致的后果其实就是少卖，这里我们 需要增加库存回退能力、定时监测能力

名额扣成功了，但是库存扣失败了

  • 相当于是 Redis 执行成功了，但是 MySQL 执行失败了，这个要看是哪种情况，如果是超卖导致，那么此时是正常的，相当于 MySQL拒绝超卖成功
  • 如果是其它异常，比如：网络波动导致访问 MySQL 失败，这种情况就会导致少卖，少卖也会有专门的比对逻辑来保障

消息队列投递消息失败怎么办

  • 如果为了削峰，在预扣逻辑和库存扣减逻辑之间，再加了一层消息队列，就要考虑如果投递消息队列失败会怎样？其实可以想一下这个问题的本质是什么？
    • 本质还是向 MySQL 传递库存扣减这条链路上失败了，所以和上面的所说的库存扣减失败，可以看作是属于一种情况，解决方案还是把它归于少卖场景，通过少卖比对逻辑来进行兜底

引入消息队列之后，如果发生多次消费怎么办

  • 这个解决思路就是幂等性考虑，通过幂等性和成功才 ACK，以达到消息队列至少一次消费的语意

如果服务执行过程中挂掉怎么办？

  • 首先我们的服务是没状态的，所以挂掉重启新的请求肯定是没问题的。关键就是进行中的请求，会怎样，分几个挂掉的关键时间节点来看：
    1. 如果是请求刚进来就挂掉，那自然是毫无影响
    2. 如果名额扣减了挂掉，那么此时库存还没扣减，表现出来的现象就是短期少卖，依赖少卖监测逻辑来纠错
    3. 如果是名额扣减了，投入消息队列之后挂掉，服务恢复之后会自动消费，不会有影响

核心就是分析挂掉之后对数据的影响，看到底是导致 超卖 还是 少卖

集群化部署

集群化部署，就是从 单点 变成 一群节点，满足这个条件，宏观上都可以叫做集群化部署

集群化是很重要的部署方式，一般来说是两个维度：

  1. 追求高可靠，就是主从部署，这种模式的核心就是备胎逻辑，比如：一个节点如果挂掉了，另一个节点顶替上来服务，这就是主从模式，如果是无数据的业务服务，主从模式是比较简单的，如果是有数据的，比如：Redis 存储、MySQL 存储，这时候通常要有数据同步的解决方案，毕竟从节点上位之后，要是之前业务数据都不在了，这是不可接受的
  2. 追求高性能，就是分片部署，单个节点无论怎么垂直扩展性能，最终都会到一个瓶颈，所以另一个思路就是单个节点的性能低就低了，一群节点平摊压力，这样整体性能不就起来了么，可以看出，比如：扩展单节点性能，这种模式的扩展性，是强的多的，甚至只要你的组件能水平扩展，性能理论上就是无限的，扛不住就加机器

存储层

MySQL

  • 首先主从模式肯定可以上的，主从模式可以保护 MySQL，如果挂节点挂掉了，从节点可以替换上
  • 另外主从模式下也可以实现读写分离，将实时性不需要那么高的读操作分流到从节点，主节点就聚焦核心写处理逻辑
  • 至于是否需要分片部署，还是要结合业务、结合请求量来分析。比如：我们秒杀场景，核心两张表，一个库存扣减表，这个表基本不会随流量膨胀，比如：做一个苹果电脑的秒杀活动，就算流量几十万每秒，那不是还是一条库存记录么?
  • 除了库存表，还有一个是秒杀记录表，实际上如果用了 Redis 预扣名额，那么此时的秒杀记录表，都是抢到名额的用户，这个数据也不至于达到百万级，所以也不需要分片部署。通过分析了秒杀的业务模式，我们也就确定了，MySQL 不需要分片部署

Redis

  • 同样的，为了安全，主从模式是需要的，主节点宕机，可以换从节点顶上
  • 同时可以开启 Redis 自带的哨兵模式，自动发现主节点宕机，以及自动替换从节点，相当于就是支持自动化切换主节点的主从模式
  • 至于是不是需要分片，这里其实是要看请求量，单个 Redis 写操作实际有几万的 QPS，在这个压力之内，不用分片如果是大于这个压力，那么是需要用分片模式，比如：一个Redis支持了几万 QPS，几十万的话就需要分片支持

SecKill 服务

  • SecKill 服务就是 秒杀服务，也就是我们的核心，因为没拆得特别细，所以商品逻辑、秒杀逻辑、订单逻辑都在我们的秒杀服务中，好在设计下这些逻辑都是无状态的，整个服务也就是无状态的
  • 无状态服务可以很容易做到多机部署，水平扩容

网关服务

  • 网关从功能上可以分为 业务网关 和 流量网关，单体项目不需要业务网关，直接在应用前加一层 Nginx 即可
  • 网关服务是在秒杀服务之前，用来进行鉴权和请求转发的。所以网关服务不止没状态，连业务逻辑几乎都没有，可以很轻松的水平扩展，也就是走分片部署，注意网关服务本身是不存储数据的，本身有分片了，也就不需要主从部署了

Nginx

  • 单个 Nginx 其实够了，作为负载均衡服务，它是非常成熟稳定的
  • 但是不要一万就怕万一，软件领域总有各式各样的理解挂掉，比如：断电重启，所以如果一定要更可靠，可以用 Keepalived + Nginx 来进行高可靠部署

依赖中间件介绍

当前服务依赖的中间件为：MySQL、Redis、Nacos、Kafka

  • MySQL：是存储，用来存储持久化数据，比如：Example 表数据
  • Redis：由于其基本都将数据维护在内存中进行操作，所以拥有数据操作快的特性，我们一般用来做缓存，缓存很好理解，查询数据时将结果临时存储在 Redis，下次查询时直接从 Redis 拿到结果即可。另外 rediscover 凭借其丰富的数据结构和性能，所以可以用来设计一些系统优化点
  • Nacos：Nacos 是用来做服务治理和服务发现的，当我们定义好一个微服务接口之后，需要完成服务注册之后，后续才能被接口调用者使用。调用者使用 feign 客户端，像调用本地方法一样调用远程的微服务接口
    1. Nacos 作为服务注册中心：
      • 服务提供者（Provider）在启动时，将自己的服务信息注册到 Nacos 注册中心，包括：服务名称、IP 地址、端口号等
      • 服务消费者（Consumer）在启动时，从 Nacos 注册中心获取服务提供者的服务信息，并将这些信息缓存到本地
    2. Feign 实现服务调用：
      • 服务消费者使用 Feign 客户端进行服务调用。Feign 是一个声明式的 Web 服务客户端，使得编写 HTTP 客户端变得更容易
      • 在服务消费者的项目中，通过定义 Feign 接口并使用注解来配置调用的服务提供者的接口。例如：使用 @FeignClient 注解来指定服务提供者的服务名称，使用 @GetMapping 或 @PostMapping 等注解来指定调用的 URL 和 HTTP 方法
      • 当服务消费者调用 Feign 接口时，Feign 会根据配置自动生成 HTTP 请求，并发送给对应的服务提供者
      • 服务提供者接收到请求后，处理请求并返回结果给服务消费者
  • Kafka：一种高性能、分布式的消息队列，一般常用的场景有：
    1. 消息传递：Kafka 可以用于在分布式系统中进行消息传递。生产者（producer）向 Kafka 的一个（或多个）topic 发送消息，消费者（consumer）订阅这些 topic 并处理接收到的消息
    2. 日志收集：Kafka 可以收集各种服务的日志，并将其统一汇总，然后实时地传输到大数据处理平台进行处理
    3. 数据集成：Kafka可以作为数据集成的一种方式，它可以连接各种数据源和数据目的地，使得数据可以在不同的系统之间流动
    4. 数据缓冲：在数据处理过程中，如果下游系统处理速度跟不上上游系统，Kafka可以起到缓冲作用，将上游产生的数据暂时存放在 Kafka 中，等待下游系统慢慢处理

V2

库存扣减性能不给力，用 Redis 预扣来加速

v1 版本的秒杀性能非常低，主要有 2 个原因：

  1. 关系型数据 MySQL 性能本身就不高，最多也就能几千 QPS
  2. 库存操作都集中在 MySQL 同一行中，在 MySQL 同一行进行并发写操作，效率会非常低，导致仅仅300 QPS

解决办法：

  • 引入 Redis 来做库存的预扣，实际就是做流量的提前筛选，在筛选中胜出的流量才有机会抵达 MySQL

Redis 预扣流程

  • 原来的流程：请求到达 → 参数检测 → 秒杀库存
  • 引入预扣之后的流程：请求到达 → 参数检测 → 秒杀名额 → 秒杀库存
  • 秒杀名额具体来说其实就是在 Redis 做了一遍名额的争抢，对 Redis 而言，和 MySQL 一样需要实现 记录库存、记录状态、用户额度限制 这一类功能，这里的设计原则是尽可能将问题在更前面暴露，即能在Redis 暴露的问题，比如：用户额度限制，就不要等传递给 MySQL 之后才发现

解决原子性问题，用Lua脚本

潜在的原子性问题

引入 Redis 来扣减名额，扣减的流程应该如下：

  • 收到请求 → 从 Redis 查看商品名额余量 → 如果余量大于 0 → 在 Redis 进行名额数量扣减 → 在 Redis 写入一条秒杀记录

会出现的问题：

  • 在很多请求并发过来的时候，可能查看名额时，名额余量大于 0，等你真的要扣减的时候，实际上已经被其它请求扣减到 0 了，也就是说，查看名额，和扣减库存，实际是存在时间 GAP 的，这样可能会导致名额超发，虽然最终流量到 MySQL 的时候可以兜底，库存不会超发

解决办法：

  • 需要原子性，要保证原子性，首先想到得是事务，Redis 提供了两种事务方式：
    1. 一种是 MULTI，但这种方式其实属于半成品，使用起来也很复杂，已经被 Redis 官方抛弃
    2. 另一种是结合 LUA 脚本，通过 Lua 脚本的整体性，加上 Redis 的单线程特性，巧妙实现了事务
  • Redis 是在 2.6 版本通过内嵌支持 Lua 环境，执行脚本的常用命令为 EVAL

实现流程：

  • 定义关键 KEY

    • key1：用户 id，表明是哪个用户
    • key2：商品 id，表明是哪件商品
    • key3：希望抢购多少个名额
    • key4：秒杀单号，作为这个秒杀行为的唯一标识
    • key5：秒杀记录，以秒杀单号为 Key 对应的秒杀详细信息，该信息实际就是一个 JSON 数据

  • 判断用户是否在秒杀中

    • 判断是否秒杀中，即查询 keyUserGoodsSecNum 这个 Key，如果有记录目前的 secNum，则在秒杀中，并且将秒杀单号返回

  • 判断用户是否超过限额

    • 判断用户在此商品是否超过限额，即查询 keyUserSecKilledNum 这个 Key，如果用户剩余额度不足，则拒绝

V3

流量削峰，给预扣和库存之间加一层

消息队列 可以优雅地解决这个问题，将双十一的请求扔入更高速的中间存储一般而言就是消息队列，然后等待后续服务慢慢处理

V2 潜在的问题

V2 中引入 Redis 之后的流程变为了：请求到达 → 参数检测 → 秒杀名额 → 秒杀库存

  • 这里会出现的问题：这里本质就是一个高速存储 Redis 和低速处理 MySQL 之间有并发能力差，Redis 很快就处理完了，即使是过滤了一波的请求也还是有可能打崩 MySQL
  • 解决方案：引入消息队列来进行消息的异步传递，这个过程实现了解耦，更重要的也实现了削峰，MySQL 可以慢慢进行消费了，而消息投递时也不会受 MySQL 性能影响了，整体的处理速度就更快了

  1. 通过预扣，选出一批拿到名额的请求，这一步已经实现了很大一部分流量的筛选
  2. 为了避免筛选出来的流量依旧超过 MySQL 的处理能力，业务层将拿到名额的秒杀请求会投递到 消息队列
  3. 库存逻辑会消费消息队列的数据，这时候的拉取频率完全可以按自己的能力来调节，基本上是可以实现均匀性消费
  4. 进行扣减操作也就是更新 DB 数据，这里稍微扩展一下，虽然削峰可以防止 MySQL 被打崩，但是如果商品数量很多，这里可能处理起来会比较慢，所以在业务规模很大的情况下，这里还有个潜在的优化点，即：批量扣减库存，在已经是削峰异步的情况下，批量聚合还是比较容易的

消息队列方案选型

  • 消息队列：顾名思义就是传递消息的队列，有着先入先出的特性，既然是队列，自然遵循先入先出的原则，同时消息队列具备 可靠性、高性能 等特点
  • 消息队列是大型分布式系统不可缺少的中间件，一般用于 异步流程、消息分发、流量削锋 等问题，可以通过消息队列实现 高性能、高可用、高扩展 的架构
  • 业界比较出名的消息队列有：ActiveMQ、RabbitMQ、ZeroMQ、Kafka、MetaMQ、RocketMQ

消息队列应用场景

  • 异步

    • 比如：视频处理，涉及到视频下载，那受限于网络带宽等因素，扩容无用；再比如一个业务流程，过了 10 多个微服务，单个也许不长，加起来就很难接受
    • 以上的情况下，用户很难通过同步接口长时间等待结果，那就应该做成异步，先扔进消息队列，后续再进行消费，响应时间可以从 10s，降低到 10ms

  • 消息分发

    • 假设一个核心服务 A，是用来发布某种信号的，发布之后，需要通知到下游服务 B、C，这种模式在只有 B、C 两兄弟的时候，没啥问题。但随着业务需要，可能会有 D、E、F 等更多的打工人出现，这时候 A 服务就需要更改代码，将消息也传递给这些新加入的兄弟，每次增加打工人，就需要更改一次代码

    • 解决办法：引入消息队列，则可以解决这个问题，实现能力复用，业务解耦

  • 流量削峰

    • 秒杀场景就需要削峰的能力，将库存操作均匀化处理，以防止瞬时流量打崩 MySQL

    • 在架构上，削峰和上面的异步场景是相同的架构，都是将请求扔入队列中，再慢慢消费

  • 但要注意异步场景更偏向于单个请求，本身处理时间很长。削峰针对是单个请求没问题，但是流量突发扛不住的场景

  • 选型的时候，需要根据业务场景，比如：要支持天猫双十一类超大型的秒杀活动，这种一锤子买卖，那管理界面、消息回溯啥的不重要。优先选 Kafka 和 RocketMQ 这种更高吞吐的。

秒杀中用哪个

  • 秒杀对性能的要求高，用高吞吐的不会有错，这里 RocketMQ 和 Kafka 都能达到我们的要求，其他功能上两者也不分伯仲，而 RocketMQ 在大量 topic 下具备一些优势，这个在我们的场景是不存在的

消息队列下实现仅消费一次语意

消息模式

  • 对于消息，一般有如下几种消费模式：
    • at most once，即最多一次语义：消息可能会丢失，但绝不会被重复发送
    • at least once，至少一次语义：消息不会丢失，但有可能被重复发送
    • exactly once，精确一次语义：消息不会丢失，也不会被重复发送
  • 消息队列一般实现得都是 at least once 语意，比如：Kafka 就是如此，也就是只要消息进入了Kafka，那么你不用担心消息会丢失，但是要考虑被重复发送的可能性
  • 实际上在后端业务中，很多时候对消费是希望能 exactly once，也就是精确一次语义，目标就是一笔操作不想造成多次影响。显然，这一点实际光靠消息队列是做不到的，需要消息队列与业务逻辑配合完成
  • 解决思路：至少一次 + 可重入消费

至少一次消费

  • Kafka 至少一次消费实际依托于如下两点：

    1. 存储数据不丢失，也就是进去消息队列之后的数据，是持久的，Kafka 这种消息队列，是非常可靠的，只要写入了队列，就不会丢消息，当然，这里还有个写入策略配置。如下表：

      acks 行为 可靠性 性能
      0 生产者在发送消息后不会等待来自服务器的确认，所以生产者实际上不知道消息是否成功，也就无法重试，生产可靠性是最低的 不可靠 最高
      1 生产者会在消息发送后等待主节点的确认，但不会等待所有副本的确认 相对可靠 还是比较高
      all 只有在所有副本都成功写入消息后，生产者才会收到确认。这确保了消息的可靠性，但延迟显然是最高的 可靠 会低一些
      • 一般情况会将 acks 设置为 1 ，丢失数据的情况会很少，几乎可以忽略，同时性能也是很高的

    2. 消费失败不清除，也就是 ACK 机制，如果有错误的消息那我们就不 ACK，这样下次还能继续拉取这条消息来消费，本质是实现了无限重试的能力

  • 通过 存储数据不丢失 + 消费失败不清除，就可以实现至少一次语意，也就保证了业务层在任何异常情况下，都可以反复执行这条消息，直到成功

可重入消费

  • 要实现至少消息一次，其实就一定需要实现重新消费，所以这里的 只消费一次，并不是说这条消息真的只做一次消费，这是保证不了的，而是最终消费造成的结果，只会产生一次影响，也就是咱们之前说的幂等性

  • Nacos（Dynamic Naming and Configuration Service）是阿里巴巴开源的一个更易于构建云原生应用的动态服务发现、配置管理和服务管理平台。随着微服务架构的兴起，服务间的通信和配置管理变得尤为重要
  • Nacos 提供了一组简单易用的特性集，包括服务发现、服务配置、服务元数据及流量管理等，帮助开发者更高效地管理和维护微服务应用

Nacos 的优势：

  1. 动态服务发现：Nacos 支持基于 DNS 和基于 RPC 的服务发现，服务提供者和消费者可以实现自动注册和发现
  2. 统一配置管理：通过 Nacos 可以统一管理不同环境、不同集群的配置，实现配置的动态更新
  3. 高可用与可扩展：Nacos 集群支持高可用部署，保障服务的稳定性和连续性。同时，其良好的扩展性可以满足不同规模的业务需求
  4. 多环境支持：无论是开发、测试还是生产环境，Nacos 都能提供一致性的服务发现和配置管理体验
  5. 集成与兼容性：Nacos 可以与其他开源生态（如：Spring Cloud、Dubbo等）无缝集成，降低了开发和运维的复杂性

服务注册与服务发现原理

在微服务架构中，服务注册与发现是实现服务间通信的关键环节。Nacos 通过以下步骤实现服务注册与服务发现：

  1. 服务注册
    • 当服务提供者启动时，会向 Nacos 注册中心发送注册请求，包含自身的元数据信息（如：服务名、IP地址、端口号等）。Nacos接收请求后，将服务提供者的信息存储在内存中，并持久化到数据库中，以确保数据的高可用性和一致性
  2. 服务发现
    • 服务消费者在启动时或运行过程中，会向 Nacos 注册中心发送服务发现请求，查询所需服务的提供者列表。Nacos 根据消费者的请求条件（如：服务名、版本号等），返回匹配的服务提供者列表。这个列表包含了提供者的网络地址和其他元数据，消费者可以根据这些信息选择合适的服务提供者进行调用
  3. 健康检查
    • 为了确保服务的可用性和稳定性，Nacos 还提供了一套健康检查机制。它会定时向服务提供者发送健康检查请求，验证其是否能够正常处理请求。如果服务提供者连续多次未通过健康检查，Nacos 会将其标记为不健康状态，并从服务提供者列表中移除。这样可以避免消费者调用已经不可用的服务提供者，提高了系统的容错能力和可用性
  4. 长连接与事件通知
    • Nacos 通过长连接（HTTP长轮询 或 WebSocket）与客户端保持通信，以实现实时的服务变更通知。当服务提供者有变更（如：新增实例、实例下线等）时，Nacos会基于长连接推送变更数据给消费者，确保消费者能够及时感知到服务的变化并作出相应的调整。这种机制保证了服务发现和配置的实时性和准确性
  5. 负载均衡与容错
    • 在服务消费者端，Nacos 提供了负载均衡策略，支持多种负载均衡算法（如：随机、轮询、加权轮询等）。消费者从提供者地址列表中，基于负载均衡算法选一台提供者进行调用。如果调用失败，会再选另一台调用，实现了容错和高可用性

总之，Nacos 通过简单易用的特性集和高效的服务注册与发现机制，为微服务架构提供了强大的支撑能力。它不仅降低了微服务应用开发和运维的复杂性，还提高了系统的可用性和稳定性。在云原生时代，Nacos 将继续发挥重要作用，助力企业构建高效、稳定的微服务应用

用 Nginx 来做网关服务的负载均衡

网关服务提供了鉴权和转发的能力，网关服务自身没状态，可以很轻松的水平扩展，所以网关服务本身需要看作由多个节点组成，在前面还需要接入一个 Nginx 来做负载均衡

Nginx

Nginx 是一个轻量级的反向代理服务器，用 Nginx 可以非常方便地为你的 Web 服务提供对外访问，即：请求先打入 Nginx，Nginx 再负载均衡到你自己的业务服务器

Nginx网络模型

网络模型有很多种，一般而言常见的有单线程、多线程、多进程这几类，单线程不用多说，下面介绍一下多线程模型和多线程模型

  • 多线程模型
    • 多线程模型，即：使用多线程来处理多个任务，在对外服务中主要还是指一个请求到来，就创建线程来进行处理的模型，当然，不见得是每个请求来都创建，可能还涉及到线程的复用
    • 知名软件 memcached 就是多线程模型。这种模型核心有如下2个优点：
      1. 速度快，线程创建成本比进程低很多
      2. 数据共享方便，多线程共享虚拟地址空间，同一进程下的多线程可以很方便共享数据
  • 多进程模型
    • 多进程模型即是用多进程来处理多个任务，对于对外服务而言，主要指当一个请求到来，这时候实际是创建进程来进行处理，最为知名的多进程模型软件就是 Nginx
    • 多进程模型具备如下优点：
      1. 编程相对容易，不需要搞成同进程下多线程竞争相同资源问题
      2. 稳定性强，一个进程崩溃是完全不会影响另一个进程的，比如：极端一点，一个进程因为内存使用不当，OOM了，此时只有这个进程被kill掉，但是如果是多线程模型下，某个线程胡来，整个进程遭殃，同时多进程很容易做到重启恢复，比如：Nginx 的 Master 发现某个 worker 进程挂掉了，直接重启就行

Nginx 为何选择多进程

  1. 瓶颈在I/O，接入层又不处理业务，主要瓶颈是I/O，而不是CPU，所以 Nginx 也用了 I/O 多路复用技术，这点和 Redis 其实有异曲同工之处
  2. 可靠性，在 Linux 上，进程的恢复比线程容易太多了，Nginx 作为入口的负载均衡服务，稳定性是一个非常大的考重

Nginx做负载均衡

  • 负载均衡 其实说简单一点，就是 提供均衡访问下游服务节点的组件，可以是软件也可以是硬件，这里 Nginx 就是软件类型的负载均衡，所谓均衡访问，其实不一定是均匀，而是按一定策略来访问，比如最常见的，就是轮询访问，就是以此访问下游节点，这种方式就非常平均
  • 但是我们实际的业务场景下，不一定是需要流量均匀打到后端，比如：后面某个节点性能更强，那么他负责更多的流量也理所当然，为了支持不同场景的需要，Nginx 一共提供了 5 种负载均衡策略
策略 说明 场景
轮询 按时间顺序均匀将请求打向后端各个节点 后端节点性能都差不多，或者不需要区分流量
加权轮询 根据权重按时间顺序将请求打向后端各个节点 后端机器性能差距比较大，能者多劳
根据服务响应 根据后端节点的响应时间来调节，响应越快收到的流量越大 务实主义，不根据机器固有性能，而是根据反馈
按 IP Hash 根据 IP Hash 将请求打向后端各个节点 希望用户落在相同节点，这种一般是针对有状态的服务
按 Url Hash 根据 Url Hash 将请求打向后端各个节点 如果后端节点有做资源缓存会比较有用，相同资源的请求都会落在同一个节点

高可靠容灾：Redis 部署主从和哨兵

容灾的必要性

  • 秒杀服务依赖了 Redis，而且它不仅用来当缓存，还用作非常关键的秒杀预扣，所以属于关键链路，是关键链路，我们就得非常严肃得考虑容灾
  • Redis 是内存数据库，虽然可以通过持久化机制，保存快照或者执行记录文件在磁盘上，等重启时供恢复，但考虑这么一个问题，如果这台机器故障了，或者磁盘坏了，那数据不就全没了吗？这种情况又该怎么办呢？

主从模式

  • 主从模式最容易想到的，就是主从模式，即：给 Redis 加上从节点，当 Master 挂掉了，可以启用从节点取而代之，这种常见模式，我们一般需要关注两个核心问题:
    1. Master 和 Slave 之间数据怎么同步？
    2. Slave 什么情况下可以升级为 Master？

Slave 数据同步

eg：

  • 怎么能知道业务连接出问题了，这里最简单的方式是 通过 监控，当发现出问题了，就进行切换
  • 如果主节点出现问题，我们通过监控发现，然后人工去进行切换，这其实很不便利，也容易让故障得不到及时修复，所以我们可以写一些脚本来进行监测，比如：在某个服务器上，专门运行脚本，检查 Redis 的状况，发生问题还是用脚本自动切换
  • 实际上，这种监控以及自动切换的能力，Redis 已经有了完整解决方案，就是 哨兵模式

哨兵模式

  • 哨兵，顾名思义，最重要的就是 放哨，即：监测 Redis 节点故障、并且提供故障转移的能力。说白了就是一套用来检测 Redis 服务运行情况的程序，发现了就做一些对应处理，这里有一点需要关注，Redis的哨兵服务，本质上也是 Redis 进程，只是启动参数不同，有不同的职责

  • 哨兵选主策略

    • 当哨兵集群选举出哨兵 Leader 后，由哨兵 Leader 从 Redis 从节点中选择一个 Redis 节点作为主节点：

      1. 过滤故障的节点
      2. 选择优先级 slave-priority 最大的从节点作为主节点，如不存在，则继续
      3. 选择复制偏移量最大的从节点作为主节点，如果都一样，则继续。这里解释下数据偏移量记录写了多少数据，主服务器会把偏移量同步给从服务器，当主从的偏移量一致，则数据是完全同步
      4. 选择 runid 最小的从节点作为主节点。Redis 每次启动的时候生成随机的 runid 作为 Redis 的标识

  • 哨兵集群中哨兵Leader的选拔

    • 首先，每一个哨兵节点都可以成为 Leader，当一个哨兵节点确认 Redis 集群的主节点主观下线后，会请求其他哨兵节点要求将自己选举为 Leader。被请求的哨兵节点如果没有同意过其他哨兵节点的选举请求，则同意该请求，也就是选举票数 + 1，否则不同意
    • 如果一个哨兵节点获得的选举票数超过节点数的一半，且大于 quorum 配置的值，则该哨兵节点选举为Leader；否则重新进行选举。特别说一下，quorum 就是一个配置值，一般配置得小于节点的一半，也就是没有特殊需要，没必要让 quorum 生效

高可靠容灾：MySQL 主从架构容灾

Redis 作为秒杀的加速器，当然是非常的重要的，但是说到重要性，实际上还是比不过作为底层存储的MySQL，毕竟 Redis 数据出问题了，最终 MySQL 都可以兜底的。而 MySQL 一旦数据丢失，那就真的是场灾难了。所以 MySQL 的容灾，势在必行

主从MySQL

  • 主从 MySQL 是指一个主节点 MySQL 提供服务，一个从节点 MySQL 同步数据作为备份

MySQL 容灾策略还分为下面的几种模式：

  1. 一主一从模式：一个大佬带一个小弟，大佬挂了小弟上位

  2. 一主多从模式：一个大佬带一群小弟，只要不全挂，就还能翻盘

  3. 级联主从模式：一个大佬培养了一个亲信骨干，其它小弟都由亲信骨干培养

主从复制原理

  • 主节点工作
    • 当主节点上进行写操作时，会按照时间先后顺序写入到 binlog 中。主从复制就是基于 binlog 进行的，具体流程有两部分：
      • 首先，当从节点连接到主节点时，主节点会创建一个叫做 dump 的线程，有多少个从节点，就会创建多少个 dump 线程
      • 然后，当主节点的 binlog 发生变化的时候，dump 线程就会通知从节点，并将相应的 binlog 内容发送给从节点
  • 从节点工作
  • 当开启主从同步的时候，从节点会创建两个线程用来完成数据同步工作：
    • I/O线程连接到主节点，主节点上的dump线程会将 binlog 的内容发送给 I/O 线程。它接收到内容后，再将其写入到本地的 relay log
    • SQL 线程读取 I/O 线程写入的 relay log，并且根据 relay log 的内容对从数据库做对应的操作

    • 合理性设计困难，需要兼顾 多次秒杀，崩溃后 id 获取等异常情况
    • 流量削峰，Redis 抢到名额之后的瞬间流量也可能打崩 MySQL，引入 Kafka削峰，并配合业务幂等性实现只消费一次能力，这里需要对消息队列做选型，并掌握选择 Kafka 的原因
    • 高并发架构设计，每一层都需要足够的扩展性设计，服务层需要做无状态设计，以便水平扩容，存储层也需要进行分片考虑

精准化设计

  • 精准化设计在秒杀系统中是比较明显的：
    • 通过 MySQL 乐观锁保证不多卖，即：使用 where 条件做了库存限制，不会将库存卖到 0 以下，多卖是严重事故，所以需要这种可靠机制来作为兜底保证
    • 最终我们怎么保证不少卖呢，这里实际上是需要通过定时任务进行扫描，发现不一致则进行库存返还，以此保证不少卖

性能调优

  • 秒杀系统中实现了如下的性能优化：
    • 使用缓存优化秒杀流程中的查询操作，秒杀操作的 QPS 提升1倍，注意是秒杀操作，这里只提升一倍是因为查询已经不是瓶颈，有新的瓶颈出现
    • 使用日志输出优化，提升4倍QPS，这点主要是控制台输出要去掉
    • 使用名额预扣，利用 Redis 的高性能做缓冲处理，解决 MySQL 性能低的问题最终请求还是会有一批落在 MySQL，这时候再使用聚合写，可以解决 MySQL 热 Key 写入问题，性能提升1倍以上
